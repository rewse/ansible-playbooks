# Home Assistant High Availability - 要件定義

## 概要

現在、fox（singleton_int1）で単一ノードとして稼働しているHome Assistantを、hotel（singleton_int2）をフェイルオーバー先とするHA構成に変更する。

## 機能要件

### FR-1: 自動フェイルオーバー

**ユーザーストーリー:** システム管理者として、foxがダウンした際に自動的にhotelでHome Assistantが起動することで、サービスの可用性を維持したい。

**受け入れ基準:**
1. WHEN foxのHome Assistantが停止する THEN hotelが自動的に検知してHome Assistantを起動する
2. WHEN フェイルオーバーが発生する THEN 管理者にメール通知が送信される
3. WHEN フェイルオーバーが完了する THEN Home AssistantのAPIが応答可能になる

### FR-2: 設定ファイルのレプリケーション

**ユーザーストーリー:** システム管理者として、foxの設定ファイルがhotelに自動的に同期されることで、フェイルオーバー時に最新の設定で起動したい。

**受け入れ基準:**
1. WHEN foxで設定ファイルが変更される THEN 1分以内にhotelに同期される
2. WHEN 同期が実行される THEN `/srv/homeassistant/config` 配下の全ファイルが同期される
3. WHEN 同期が実行される THEN `.storage/` ディレクトリも含めて同期される

### FR-3: ヘルスチェック

**ユーザーストーリー:** システム管理者として、foxのHome Assistantの状態を定期的に監視することで、異常を早期に検知したい。

**受け入れ基準:**
1. WHEN ヘルスチェックが実行される THEN foxのHome Assistant APIに接続して応答を確認する
2. WHEN APIチェックが失敗する THEN 30秒待機して再チェックする
3. WHEN 2回連続でAPIチェックが失敗する THEN SSH経由でfoxのコンテナ状態を確認する
4. WHEN コンテナが停止している THEN hotelでHome Assistantを起動する

### FR-4: 手動復旧

**ユーザーストーリー:** システム管理者として、foxが復旧した際に手動でfoxに切り戻すことで、通常の構成に戻したい。

**受け入れ基準:**
1. WHEN foxが復旧する THEN 管理者が手動でhotelのコンテナを停止できる
2. WHEN hotelのコンテナが停止する THEN foxのコンテナを起動できる
3. WHEN 設定ファイルに差分がある THEN 手動でマージできる

## 非機能要件

### NFR-1: 可用性

- **目標RPO（Recovery Point Objective）:** 1分（設定ファイルの同期間隔）
- **目標RTO（Recovery Time Objective）:** 2分以内（検知30秒×2回 + 起動時間）
- **稼働率目標:** 99.9%（年間ダウンタイム8.76時間以内）

### NFR-2: パフォーマンス

- **ヘルスチェック間隔:** 30秒
- **レプリケーション間隔:** 1分
- **API応答時間:** 5秒以内

### NFR-3: セキュリティ

- **rsync通信:** ローカルネットワーク内のみ（認証なし）
- **SSH通信:** 公開鍵認証
- **メール通知:** 既存のpostfix設定を利用

### NFR-4: 保守性

- **設定管理:** Ansibleで全ての設定を管理
- **ログ:** systemdのjournalに記録
- **ドキュメント:** 運用手順とトラブルシューティングを文書化

## 制約条件

### C-1: インフラストラクチャ

- alfaはVPN先で信頼性が低いため監視ノードとして使用しない
- guppyは今年廃止予定のため共有ストレージとして使用しない
- hotelは他のサービスも稼働しているため、Home Assistantのみを選択的にフェイルオーバー

### C-2: データベース

- 履歴データベースはAurora（PostgreSQL）を使用
- データベースのHA化は対象外

### C-3: 運用

- フェイルオーバー中はhotelで設定変更しない
- 復旧時は手動でfoxに戻す
- 設定ファイルの差分は手動でマージ

## 対象外

以下は本プロジェクトの対象外とする：

- データベースのHA化
- アクティブ/アクティブ構成
- 自動フェイルバック（foxへの自動切り戻し）
- 設定ファイルの双方向同期
- ロードバランシング
- 3ノード以上のクラスタ構成
